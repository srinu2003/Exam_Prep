Unit I

PL/SQL Basics

The anonymous block as PL/SQL's atomic unit, how variables behave inside it, the scalar/composite type system, control flow, structured exception handling, bulk operations, a first look at the modular building blocks, and transaction scope.

1.1 Block Structure

Every PL/SQL unit is a block with up to four sections: an optional header, an optional DECLARE section, a mandatory BEGIN...END executable section, and an optional EXCEPTION section. Blocks nest freely — a sub-block can declare its own variables that shadow the outer scope.

01_block_structure.sql
DECLARE
  v_dept_count NUMBER;
BEGIN
  SELECT COUNT(*) INTO v_dept_count FROM departments;
  DBMS_OUTPUT.PUT_LINE('Departments: ' || v_dept_count);

  DECLARE  -- nested sub-block, its own scope
    v_msg VARCHAR2(50) := 'Nested block executing';
  BEGIN
    DBMS_OUTPUT.PUT_LINE(v_msg);
  END;
EXCEPTION
  WHEN OTHERS THEN
    DBMS_OUTPUT.PUT_LINE('Error: ' || SQLERRM);
END;
/
Departments: 0
Nested block executing
SQL> PL/SQL procedure successfully completed.
Fig. 2 — Anatomy of a PL/SQL Block & Nested Scope

1.2 Behavior of Variables in Blocks

Variables declared in an outer block are visible to nested blocks, but a variable of the same name in a nested block shadows the outer one for that block's lifetime only. Variables cease to exist once their declaring block ends — nothing survives across separate anonymous block executions unless it's a package-level variable (Unit IV).

02_variable_scope.sql
DECLARE
  v_salary NUMBER := 50000;
BEGIN
  DBMS_OUTPUT.PUT_LINE('Outer salary: ' || v_salary);
  DECLARE
    v_salary NUMBER := v_salary * 1.10;  -- shadows outer v_salary
  BEGIN
    DBMS_OUTPUT.PUT_LINE('Inner (10% raise): ' || v_salary);
  END;
  DBMS_OUTPUT.PUT_LINE('Outer unaffected: ' || v_salary);
END;
/
Outer salary: 50000
Inner (10% raise): 55000
Outer unaffected: 50000
SQL> PL/SQL procedure successfully completed.

1.3 Basic Scalar and Composite Data Types

Scalar types hold a single value (NUMBER, VARCHAR2, DATE, BOOLEAN, %TYPE anchored types). Composite types hold multiple values under one name — RECORDs (including %ROWTYPE) and collections (covered in Unit II).

03_data_types.sql
DECLARE
  -- scalar, anchored to the column so it never drifts out of sync
  v_name    employees.emp_name%TYPE;
  v_salary  NUMBER(10,2) := 0;
  v_hired   BOOLEAN := FALSE;

  -- composite: %ROWTYPE record mirrors the whole row
  v_emp_rec employees%ROWTYPE;

  -- composite: user-defined RECORD
  TYPE t_dept_summary IS RECORD (
    dept_name  departments.dept_name%TYPE,
    headcount  NUMBER
  );
  v_summary t_dept_summary;
BEGIN
  SELECT * INTO v_emp_rec FROM employees WHERE emp_id = 101;
  v_summary.dept_name := 'Engineering';
  v_summary.headcount := 12;
  DBMS_OUTPUT.PUT_LINE(v_summary.dept_name || ': ' || v_summary.headcount);
END;
/
Engineering: 12
SQL> PL/SQL procedure successfully completed.

1.4 Control Structures

PL/SQL provides IF/ELSIF/ELSE, CASE, and three loop forms (LOOP, WHILE, FOR) — detailed further with iteration patterns in Unit II. Here's a salary-band classifier combining IF and CASE.

04_control_structures.sql
DECLARE
  v_salary NUMBER := 72000;
  v_band   VARCHAR2(10);
BEGIN
  v_band := CASE
              WHEN v_salary >= 100000 THEN 'A'
              WHEN v_salary >= 60000  THEN 'B'
              ELSE 'C'
            END;

  IF v_band = 'A' THEN
    DBMS_OUTPUT.PUT_LINE('Senior band');
  ELSIF v_band = 'B' THEN
    DBMS_OUTPUT.PUT_LINE('Mid band');
  ELSE
    DBMS_OUTPUT.PUT_LINE('Entry band');
  END IF;
END;
/
Mid band
SQL> PL/SQL procedure successfully completed.

1.5 Exceptions

PL/SQL separates error handling from logic via the EXCEPTION section. Handlers can catch predefined exceptions (NO_DATA_FOUND, TOO_MANY_ROWS), custom named exceptions declared with EXCEPTION_INIT, or user-defined ones raised with RAISE_APPLICATION_ERROR.

05_exceptions.sql
DECLARE
  v_salary        employees.salary%TYPE;
  e_negative_pay  EXCEPTION;
BEGIN
  SELECT salary INTO v_salary FROM employees WHERE emp_id = 9999;

  IF v_salary < 0 THEN
    RAISE e_negative_pay;
  END IF;

EXCEPTION
  WHEN NO_DATA_FOUND THEN
    DBMS_OUTPUT.PUT_LINE('No employee with that ID.');
  WHEN e_negative_pay THEN
    RAISE_APPLICATION_ERROR(-20001, 'Salary cannot be negative.');
  WHEN OTHERS THEN
    DBMS_OUTPUT.PUT_LINE('Unexpected: ' || SQLCODE || ' - ' || SQLERRM);
END;
/
No employee with that ID.
SQL> PL/SQL procedure successfully completed.
Design tip: catch specific exceptions before WHEN OTHERS — the first matching handler wins and evaluation stops there.

1.6 Bulk Operations

BULK COLLECT and FORALL replace row-by-row processing with set-based operations, cutting context switches between the PL/SQL and SQL engines. A one-time raise applied to an entire department is a classic case.

06_bulk_ops.sql
DECLARE
  TYPE t_id_tab IS TABLE OF employees.emp_id%TYPE;
  v_ids t_id_tab;
BEGIN
  SELECT emp_id BULK COLLECT INTO v_ids
  FROM employees WHERE dept_id = 10;

  FORALL i IN v_ids.FIRST..v_ids.LAST
    UPDATE employees
    SET salary = salary * 1.05
    WHERE emp_id = v_ids(i);

  DBMS_OUTPUT.PUT_LINE(SQL%ROWCOUNT || ' rows given a raise.');
  COMMIT;
END;
/
0 rows given a raise.
SQL> PL/SQL procedure successfully completed.
Why it matters: a plain FOR loop with row-by-row UPDATE does one context switch per row; FORALL does one switch for the whole batch.

1.7 Functions, Procedures, and Packages — First Look

A function always returns a value and can be used in a SQL expression; a procedure performs an action and returns via OUT parameters if needed; a package groups related functions/procedures under one name. Full architecture is in Units III–IV — here's the shape of each.

07_fpp_preview.sql
CREATE OR REPLACE FUNCTION get_annual_pay(p_emp_id IN NUMBER)
  RETURN NUMBER
IS
  v_salary NUMBER;
BEGIN
  SELECT salary * 12 INTO v_salary FROM employees WHERE emp_id = p_emp_id;
  RETURN v_salary;
END get_annual_pay;
/

CREATE OR REPLACE PROCEDURE give_raise(p_emp_id IN NUMBER, p_pct IN NUMBER)
IS
BEGIN
  UPDATE employees SET salary = salary * (1 + p_pct/100) WHERE emp_id = p_emp_id;
END give_raise;
/

1.8 Transaction Scope

A transaction starts implicitly with the first DML and ends at COMMIT, ROLLBACK, or an uncommitted DDL. SAVEPOINT lets you roll back part of a transaction. This matters most inside procedures that touch multiple tables — you want all-or-nothing behavior.

08_transaction_scope.sql
BEGIN
  UPDATE employees SET salary = salary * 1.10 WHERE dept_id = 10;
  SAVEPOINT after_raise;

  UPDATE products SET qty_on_hand = qty_on_hand - 10000 WHERE product_id = 1;

  IF (SELECT qty_on_hand FROM products WHERE product_id = 1) < 0 THEN
    ROLLBACK TO after_raise;  -- undo only the stock update
    DBMS_OUTPUT.PUT_LINE('Stock update reverted; raise kept.');
  END IF;

  COMMIT;
END;
/
Stock update reverted; raise kept.
SQL> PL/SQL procedure successfully completed.
Unit II

Language Fundamentals & Control Structures

The vocabulary PL/SQL is built from, its full iteration and cursor toolkit, and the collection types that let a block hold more than one row of data at once.

2.1 Lexical Units

The four lexical building blocks are identifiers (variable/object names), literals (fixed values like 'ACTIVE' or 42), delimiters (symbols like :=, ||, %), and comments (-- single line, /* */ multi-line).

09_lexical_units.sql
DECLARE
  v_status   CONSTANT VARCHAR2(6) := 'ACTIVE';  -- identifier, literal, delimiter
  v_full_name VARCHAR2(80);
BEGIN
  /* concatenation operator || is a delimiter,
     the strings are literals */
  v_full_name := 'Emp status: ' || v_status;
  DBMS_OUTPUT.PUT_LINE(v_full_name);
END;
/
Emp status: ACTIVE
SQL> PL/SQL procedure successfully completed.

2.2 Variables and Data Types (Revisited)

Beyond scalar/composite types, PL/SQL supports %TYPE/%ROWTYPE anchoring (seen in Unit I) plus reference types (REF CURSOR) and LOB types (CLOB, BLOB). Anchoring is the habit worth keeping — it means a column-width change never breaks your code.

10_variables_types.sql
DECLARE
  v_notes  CLOB;
  TYPE t_ref_cur IS REF CURSOR;
  v_cur    t_ref_cur;
  v_name   employees.emp_name%TYPE;
BEGIN
  v_notes := 'Onboarding complete.';
  OPEN v_cur FOR SELECT emp_name FROM employees WHERE ROWNUM = 1;
  FETCH v_cur INTO v_name;
  CLOSE v_cur;
  DBMS_OUTPUT.PUT_LINE('First employee: ' || v_name);
END;
/
First employee:
SQL> PL/SQL procedure successfully completed.

2.3 Conditional Statements

Beyond simple IF, the searched CASE statement handles multi-branch logic without nested ELSIF chains — useful for classification logic like a discount tier.

    WHEN v_qty >= 100 THEN DBMS_OUTPUT.PUT_LINE('8% bulk discount');
    ELSE DBMS_OUTPUT.PUT_LINE('No discount');
  END CASE;
END;
/
8% bulk discount

2.4 Iterative Statements

Three loop forms: basic LOOP...EXIT WHEN, WHILE (test before each pass), and FOR (fixed range, counter managed automatically). A cursor FOR loop — shown next — is the idiomatic way to iterate result sets.

12_loops.sql
BEGIN
  -- FOR loop: fixed range
  FOR i IN 1..3 LOOP
    DBMS_OUTPUT.PUT_LINE('Pass ' || i);
  END LOOP;

  -- WHILE loop: condition checked up front
  DECLARE v_n NUMBER := 0;
  BEGIN
    WHILE v_n < 3 LOOP
      v_n := v_n + 1;
    END LOOP;
    DBMS_OUTPUT.PUT_LINE('WHILE finished at ' || v_n);
  END;

  -- Basic LOOP: explicit EXIT
  DECLARE v_n NUMBER := 0;
  BEGIN
    LOOP
      v_n := v_n + 1;
      EXIT WHEN v_n = 3;
    END LOOP;
    DBMS_OUTPUT.PUT_LINE('Basic LOOP finished at ' || v_n);
  END;
END;
/
Pass 1
Pass 2
Pass 3
WHILE finished at 3
Basic LOOP finished at 3
SQL> PL/SQL procedure successfully completed.

2.5 Cursor Structures

Implicit cursors handle single-row SQL automatically. Explicit cursors (DECLARE...OPEN...FETCH...CLOSE) give control over multi-row processing; a cursor FOR loop does all that management for you, and parameterized cursors accept arguments like a function.

13_cursors.sql
DECLARE
  CURSOR c_by_dept(p_dept NUMBER) IS
    SELECT emp_name, salary FROM employees WHERE dept_id = p_dept ORDER BY salary DESC;
BEGIN
  FOR r IN c_by_dept(10) LOOP   -- cursor FOR loop: opens, fetches, closes automatically
    DBMS_OUTPUT.PUT_LINE(r.emp_name || ' - ' || r.salary);
  END LOOP;
END;
/
SQL> PL/SQL procedure successfully completed.
Cursor AttributeMeaning
%FOUNDTRUE if the last fetch returned a row
%NOTFOUNDTRUE if the last fetch returned no row
%ROWCOUNTNumber of rows fetched so far
%ISOPENTRUE if the cursor is currently open

2.6 Bulk Statements (Cursor-Driven)

Combining BULK COLLECT with a cursor and the LIMIT clause processes large result sets in manageable chunks instead of loading everything into memory at once.

14_bulk_limit.sql
DECLARE
  CURSOR c_emps IS SELECT emp_id FROM employees;
  TYPE t_id_tab IS TABLE OF employees.emp_id%TYPE;
  v_ids t_id_tab;
BEGIN
  OPEN c_emps;
  LOOP
    FETCH c_emps BULK COLLECT INTO v_ids LIMIT 100;  -- 100 rows per batch
    FORALL i IN 1..v_ids.COUNT
      UPDATE employees SET job_title = job_title WHERE emp_id = v_ids(i);
    EXIT WHEN c_emps%NOTFOUND;
  END LOOP;
  CLOSE c_emps;
END;
/
SQL> PL/SQL procedure successfully completed.

2.8 Object Types: Varray and Table Collections

A Varray is declared with a max size and stays dense. A Nested Table has no upper bound and can be stored as its own DB object type. Both can back a real column.

15_varray_nested_table.sql
-- Varray: fixed max, good for a bounded list like quarterly sales
CREATE OR REPLACE TYPE t_quarterly_sales AS VARRAY(4) OF NUMBER(10,2);
/

-- Nested Table: unbounded, good for a product's tags
CREATE OR REPLACE TYPE t_tag_list AS TABLE OF VARCHAR2(30);
/

DECLARE
  v_sales t_quarterly_sales := t_quarterly_sales(12000, 15500, 9800, 21000);
  v_tags  t_tag_list := t_tag_list('electronics', 'clearance');
BEGIN
  v_tags.EXTEND;                     -- nested table grows freely
  v_tags(v_tags.LAST) := 'featured';
  FOR i IN 1..v_sales.COUNT LOOP
    DBMS_OUTPUT.PUT_LINE('Q' || i || ': ' || v_sales(i));
  END LOOP;
  DBMS_OUTPUT.PUT_LINE('Tags: ' || v_tags.COUNT);
END;
/
Q1: 12000
Q2: 15500
Q3: 9800
Q4: 21000
Tags: 3
SQL> PL/SQL procedure successfully completed.

2.9 Associative Arrays

Also called index-by tables, these exist only in memory and can be indexed by either PLS_INTEGER or VARCHAR2 — a string-keyed lookup table is the classic use case, like caching a product's price by its SKU.

16_associative_arrays.sql
DECLARE
  TYPE t_price_map IS TABLE OF NUMBER INDEX BY VARCHAR2(30);
  v_prices t_price_map;
BEGIN
  v_prices('SKU-100') := 499.00;
  v_prices('SKU-200') := 129.50;

  DBMS_OUTPUT.PUT_LINE('SKU-100 price: ' || v_prices('SKU-100'));

  DECLARE v_key VARCHAR2(30) := v_prices.FIRST;
  BEGIN
    WHILE v_key IS NOT NULL LOOP
      DBMS_OUTPUT.PUT_LINE(v_key || ' => ' || v_prices(v_key));
      v_key := v_prices.NEXT(v_key);
    END LOOP;
  END;
END;
/
SKU-100 price: 499
SKU-100 => 499
SKU-200 => 129.5
SQL> PL/SQL procedure successfully completed.

The built-in methods every collection understands, called with dot notation.

DECLARE
  TYPE t_ids IS TABLE OF NUMBER;

BEGIN
  v_ids.DELETE(2);  -- creates a sparse collection: index 2 is gone
  DBMS_OUTPUT.PUT_LINE('Exists(2)? ' || CASE WHEN v_ids.EXISTS(2) THEN 'Y' ELSE 'N' END);

Unit III

Functions and Procedures

The full architecture of named PL/SQL program units, every notation style for calling them, and the pass-by-value vs pass-by-reference distinction that decides whether a subroutine can affect a caller's variables directly.

3.1 Function & Procedure Architecture, Transaction Scope

A named subroutine is compiled once and stored in the database as its own object (unlike an anonymous block). Its structure mirrors a block: header (name + parameters, plus RETURN type for functions), declaration section, body, and optional exception section. Like any DML inside a block, statements inside a subroutine participate in the caller's transaction unless the subroutine is marked PRAGMA AUTONOMOUS_TRANSACTION, which starts an independent transaction that commits or rolls back on its own.

18_autonomous_txn.sql
CREATE OR REPLACE PROCEDURE log_audit(p_table VARCHAR2, p_op VARCHAR2)
IS
  PRAGMA AUTONOMOUS_TRANSACTION;  -- commits independently of the caller
BEGIN
  INSERT INTO audit_log(table_name, operation) VALUES(p_table, p_op);
  COMMIT;
END log_audit;
/
SQL> Procedure created.
Why it matters: without the pragma, a ROLLBACK in the caller would also erase the audit row you're trying to guarantee gets written.

3.2 Calling Subroutines — Every Notation Style

Oracle supports five ways to pass arguments to a subroutine.

19_notation_styles.sql
CREATE OR REPLACE PROCEDURE hire_employee(
  p_name    IN VARCHAR2,
  p_dept    IN NUMBER DEFAULT 10,
  p_title   IN VARCHAR2 DEFAULT 'Associate',
  p_salary  IN NUMBER DEFAULT 45000
) IS BEGIN
  INSERT INTO employees(emp_id, emp_name, dept_id, job_title, salary)
  VALUES(employees_seq.NEXTVAL, p_name, p_dept, p_title, p_salary);
END;
/

BEGIN
  -- Positional: order must match the header exactly
  hire_employee('A. Rao', 20, 'Analyst', 52000);

  -- Named: order-independent, self-documenting
  hire_employee(p_name => 'B. Iyer', p_salary => 60000, p_dept => 30);

  -- Mixed: positional first, named after
  hire_employee('C. Verma', p_title => 'Lead');

  -- Exclusionary: simply omit a parameter that has a DEFAULT
  hire_employee('D. Singh');  -- dept/title/salary all fall back to defaults
END;
/

-- SQL Call Notation: invoking a function directly from a SQL statement
SELECT emp_name, get_annual_pay(emp_id) AS annual_pay
FROM employees WHERE dept_id = 20;
SQL> Procedure created.
SQL> PL/SQL procedure successfully completed.

3.3 Function Model Choices

A function isn't always a simple "one call in, one value out" — Oracle offers three shapes depending on what the caller needs. A scalar (normal) function returns one value per call. A pipelined table function streams rows back as they're produced, so it can be queried like a table with TABLE(...) — useful when transforming or generating large result sets. An aggregate function (built with ODCIAggregate interfaces) participates in GROUP BY like SUM or AVG. The choice depends on whether the result is a single scalar, a row set, or a per-group aggregate.

Fig. 4 — Function Model Choices
19b_pipelined_function.sql
CREATE OR REPLACE TYPE t_num_row IS TABLE OF NUMBER;
/

-- pipelined table function: streams each department's total payroll
CREATE OR REPLACE FUNCTION dept_payroll_totals RETURN t_num_row PIPELINED IS
BEGIN
  FOR r IN (SELECT dept_id, SUM(salary) AS total FROM employees GROUP BY dept_id) LOOP
    PIPE ROW(r.total);
  END LOOP;
  RETURN;
END;
/

-- query it exactly like a table
SELECT * FROM TABLE(dept_payroll_totals());
SQL> Function created.

3.4 Functions — Creation Options, Pass-by-Value vs Pass-by-Reference

A function's parameters can be IN (pass-by-value — default for scalars), OUT, or IN OUT. Scalar IN parameters are copied in (pass-by-value); large structures like records or collections can be passed with NOCOPY to hint the compiler toward pass-by-reference for performance. Creation options include DETERMINISTIC (safe to cache — same inputs always give same output) and PARALLEL_ENABLE.

20_function_options.sql
-- Pass-by-value IN parameter; DETERMINISTIC lets Oracle cache repeat calls
CREATE OR REPLACE FUNCTION apply_tax(p_amount IN NUMBER)
  RETURN NUMBER DETERMINISTIC
IS BEGIN
  RETURN ROUND(p_amount * 1.18, 2);
END;
/

-- NOCOPY hints pass-by-reference for a large collection parameter
CREATE OR REPLACE FUNCTION total_of(p_amounts IN NOCOPY t_id_list)
  RETURN NUMBER
IS
  v_total NUMBER := 0;
BEGIN
  FOR i IN 1..p_amounts.COUNT LOOP
    v_total := v_total + p_amounts(i);
  END LOOP;
  RETURN v_total;
END;
/
SQL> Function created. (x2)
NOCOPY is a hint, not a guarantee — Oracle may still copy the parameter if an exception could leave it in an inconsistent state.

3.5 Procedures — Pass-by-Value vs Pass-by-Reference

IN parameters are pass-by-value (the procedure gets a copy). OUT and IN OUT parameters behave like pass-by-reference — changes are written back to the caller's variable when the procedure completes successfully.

Fig. 5 — Parameter Modes: Value vs Reference Flow
21_procedure_params.sql
CREATE OR REPLACE PROCEDURE process_order(
  p_qty        IN NUMBER,               -- pass-by-value: copy in
  p_unit_price IN NUMBER,
  p_total      OUT NUMBER,             -- pass-by-reference: written back on exit
  p_status     IN OUT VARCHAR2          -- pass-by-reference: read AND written
) IS BEGIN
  p_total  := p_qty * p_unit_price;
  p_status := p_status || ' -> PROCESSED';
END;
/

DECLARE
  v_total  NUMBER;
  v_status VARCHAR2(40) := 'NEW';
BEGIN
  process_order(5, 199.99, v_total, v_status);
  DBMS_OUTPUT.PUT_LINE('Total: ' || v_total || ' | Status: ' || v_status);
END;
/
Total: 999.95 | Status: NEW -> PROCESSED
SQL> PL/SQL procedure successfully completed.

Unit IV

Packages

Grouping related logic into a single named, versioned unit — built around an inventory-management package so the specification/body split and rights model have a real workload behind them.

4.1 Package Architecture

A package always has two parts compiled and stored separately: the specification (public interface — what callers see) and the body (private implementation). The body can change without invalidating objects that depend only on the spec, as long as the spec itself doesn't change.

Fig. 6 — Package Specification vs Package Body

4.2 Package Specification — Prototype Features, Serially Reusable, Variables & Types

The spec declares public constants, variables, types, exceptions, and subprogram prototypes (signature only — no body). PRAGMA SERIALLY_REUSABLE tells Oracle to reset package state after each call instead of keeping it for the whole session — useful for packages that shouldn't leak state between unrelated callers.

23_inventory_pkg_spec.sql
CREATE OR REPLACE PACKAGE inventory_pkg IS

  -- PRAGMA SERIALLY_REUSABLE;  -- uncomment to reset state after every call

  -- public type & variable
  TYPE t_stock_rec IS RECORD (product_id NUMBER, qty NUMBER);
  g_low_stock_threshold CONSTANT NUMBER := 20;

  -- public exception
  e_insufficient_stock EXCEPTION;

  -- subprogram prototypes only -- bodies live in the package body
  FUNCTION get_stock(p_product_id NUMBER) RETURN NUMBER;
  PROCEDURE restock(p_product_id NUMBER, p_qty NUMBER);
  PROCEDURE ship_order(p_product_id NUMBER, p_qty NUMBER);

END inventory_pkg;

4.3 Package Body — Implementation of Every Component

The body supplies the full implementation for each prototype and may add its own private variables, types, and helper subprograms invisible to callers.

24_inventory_pkg_body.sql
CREATE OR REPLACE PACKAGE BODY inventory_pkg IS

  -- private helper, not visible outside this body
  PROCEDURE log_movement(p_product_id NUMBER, p_op VARCHAR2) IS
  BEGIN
    INSERT INTO audit_log(table_name, operation, new_value)
    VALUES('PRODUCTS', p_op, 'product_id=' || p_product_id);
  END;

  FUNCTION get_stock(p_product_id NUMBER) RETURN NUMBER IS
    v_qty NUMBER;
  BEGIN
    SELECT qty_on_hand INTO v_qty FROM products WHERE product_id = p_product_id;
    RETURN v_qty;
  END;

  PROCEDURE restock(p_product_id NUMBER, p_qty NUMBER) IS BEGIN
    UPDATE products SET qty_on_hand = qty_on_hand + p_qty WHERE product_id = p_product_id;
    log_movement(p_product_id, 'RESTOCK');
  END;

  PROCEDURE ship_order(p_product_id NUMBER, p_qty NUMBER) IS
    v_available NUMBER;
  BEGIN
    v_available := get_stock(p_product_id);
    IF v_available < p_qty THEN
      RAISE e_insufficient_stock;
    END IF;
    UPDATE products SET qty_on_hand = qty_on_hand - p_qty WHERE product_id = p_product_id;
    log_movement(p_product_id, 'SHIP');
    IF get_stock(p_product_id) < g_low_stock_threshold THEN
      DBMS_OUTPUT.PUT_LINE('Warning: product ' || p_product_id || ' is low on stock.');
    END IF;
  EXCEPTION
    WHEN e_insufficient_stock THEN
      RAISE_APPLICATION_ERROR(-20010, 'Not enough stock to ship this order.');
  END;

END inventory_pkg;
/

4.4 Definer Rights vs Invoker Rights

By default a package runs with definer rights — it executes with the privileges of whoever owns it, regardless of who calls it. Adding AUTHID CURRENT_USER switches to invoker rights — the package runs with the caller's own privileges, and unqualified table references resolve in the caller's schema. Payroll code that must always touch a fixed schema uses definer rights; a shared utility meant to act on each caller's own data uses invoker rights.

25_invoker_rights.sql
CREATE OR REPLACE PACKAGE payroll_utils AUTHID CURRENT_USER IS
  FUNCTION my_dept_total RETURN NUMBER;  -- resolves EMPLOYEES against the caller's own schema
END payroll_utils;

4.6 Comparing Validation Methods: Timestamp vs Signature

Oracle can revalidate dependent objects using either method. Timestamp mode invalidates any dependent whenever the referenced object recompiles, even if nothing meaningful changed. Signature mode only invalidates dependents if the actual call signature (parameter names, types, order) changed — recompiling a package body that keeps the same spec signature won't cascade-invalidate its callers.

Unit V

Triggers

Event-driven PL/SQL blocks that fire on DDL, DML, or system/database events — built around audit_log so every trigger in this unit produces a real, inspectable audit trail.

5.1 Introduction to Triggers & Database Trigger Architecture

A trigger is stored PL/SQL that fires automatically in response to a timing (BEFORE/AFTER/INSTEAD OF), an event (INSERT/UPDATE/DELETE/DDL/system event), and a target (table, view, schema, or database). It has no parameters and cannot be called directly — the event is its only invocation path.

Fig. 8 — Trigger Firing Sequence for an UPDATE Touching 3 Rows

5.2 DDL Triggers — Event Attribute Functions & Building One

DDL triggers fire on CREATE, ALTER, DROP, etc. Inside the trigger body, event attribute functions like ORA_SYSEVENT, ORA_DICT_OBJ_NAME, and ORA_DICT_OBJ_TYPE report exactly what happened.

27_ddl_trigger.sql
CREATE OR REPLACE TRIGGER trg_audit_ddl
  BEFORE DDL ON SCHEMA
DECLARE
BEGIN
  INSERT INTO audit_log(table_name, operation, new_value)
  VALUES(ORA_DICT_OBJ_NAME, ORA_SYSEVENT, 'obj_type=' || ORA_DICT_OBJ_TYPE);
END;

5.3 DML Triggers — Statement-Level vs Row-Level

A statement-level trigger fires once per triggering statement regardless of rows affected. A row-level trigger (FOR EACH ROW) fires once per affected row and exposes :OLD and :NEW pseudorecords — essential for column-level auditing.

28_row_level_trigger.sql
CREATE OR REPLACE TRIGGER trg_salary_audit
  AFTER UPDATE OF salary ON employees
  FOR EACH ROW              -- row-level: fires per changed row
BEGIN
  INSERT INTO audit_log(table_name, operation, old_value, new_value)
  VALUES('EMPLOYEES', 'UPDATE', :OLD.salary, :NEW.salary);
END;
/

CREATE OR REPLACE TRIGGER trg_bulk_update_notice
  AFTER UPDATE ON employees   -- statement-level: fires once total
BEGIN
  DBMS_OUTPUT.PUT_LINE('An UPDATE statement touched the EMPLOYEES table.');
END;

5.4 Compound Triggers

A compound trigger combines all four timing points — BEFORE STATEMENT, BEFORE EACH ROW, AFTER EACH ROW, AFTER STATEMENT — in a single trigger body sharing state, which is the standard fix for the mutating-table error you'd otherwise hit when a row-level trigger tries to query the table it's firing on.

29_compound_trigger.sql
CREATE OR REPLACE TRIGGER trg_dept_salary_cap
  FOR UPDATE OF salary ON employees
COMPOUND TRIGGER

  v_dept_total NUMBER := 0;

  BEFORE STATEMENT IS BEGIN
    v_dept_total := 0;
  END BEFORE STATEMENT;

  AFTER EACH ROW IS BEGIN
    v_dept_total := v_dept_total + :NEW.salary;  -- safe: reads the shared variable, not the table
  END AFTER EACH ROW;

  AFTER STATEMENT IS BEGIN
    IF v_dept_total > 1000000 THEN
      DBMS_OUTPUT.PUT_LINE('Warning: this batch pushed payroll over 10L.');
    END IF;
  END AFTER STATEMENT;

END trg_dept_salary_cap;
/
SQL> Trigger created.

5.5 INSTEAD OF Triggers

INSTEAD OF triggers make a normally non-updatable view (e.g. one joining multiple tables) accept INSERT/UPDATE/DELETE by intercepting the operation and redirecting it to the underlying base tables.

30_instead_of_trigger.sql
CREATE OR REPLACE VIEW v_employee_directory AS
  SELECT e.emp_id, e.emp_name, d.dept_name
  FROM employees e JOIN departments d ON e.dept_id = d.dept_id;

CREATE OR REPLACE TRIGGER trg_directory_insert
  INSTEAD OF INSERT ON v_employee_directory
  FOR EACH ROW
DECLARE
  v_dept_id departments.dept_id%TYPE;
BEGIN
  SELECT dept_id INTO v_dept_id FROM departments WHERE dept_name = :NEW.dept_name;
  INSERT INTO employees(emp_id, emp_name, dept_id)
  VALUES(:NEW.emp_id, :NEW.emp_name, v_dept_id);
END;
/

5.7 Trigger Restrictions — Maximum Size, SQL Statements, LONG and LONG RAW

Practical limits and legacy-datatype constraints to design around:

RestrictionDetail
Maximum trigger size32 KB of compiled code — split large logic into a called procedure/package instead
Transaction controlCOMMIT/ROLLBACK/SAVEPOINT are not allowed inside a row-level trigger body (except via an autonomous transaction)
Mutating tablesA row-level trigger cannot query or modify the table it's firing on directly — use a compound trigger instead
LONG / LONG RAWThese legacy types cannot be referenced in a trigger's :NEW/:OLD — migrate to CLOB/BLOB, which triggers can reference freely
Common pitfall: a row-level trigger on employees that runs SELECT ... FROM employees raises ORA-04091: table is mutating. Restructure as a compound trigger (Unit 5.4) so the aggregation happens in the shared package-like state instead of a live query against the firing table.
